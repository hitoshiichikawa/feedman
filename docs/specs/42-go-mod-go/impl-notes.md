# 実装メモ: Issue #42 go.mod の Go バージョン指定をメジャー.マイナーに丸める

## 変更内容のサマリ

- `go.mod` の 3 行目 `go 1.25.1` を `go 1.25`（メジャー.マイナー形式）へ丸めた。
- 変更ファイルは `go.mod` の 1 行のみ（`require` ブロック・`go.sum` は変更なし）。
- `.github/workflows/ci.yml` は変更なし（後述）。

## `toolchain` 併記の採否判断と根拠

**判断: `toolchain` ディレクティブは併記しない（メジャー.マイナー丸めのみ）。**

根拠:

- 本 Issue の目的は「当該パッチ未満のツールチェインを締め出さない」という環境制約の緩和。
  `toolchain go1.25.1` を併記するとパッチを下限固定してしまい、目的と相反する。
- CLAUDE.md / README は「Go 1.25」を前提としており、特定パッチへの固定要求は確認されていない。
- requirements.md Requirement 3（toolchain 併記時の互換性）は「併記しても破綻しない」ことを
  境界値として保証する条件付き要件であり、併記の採用を必須化していない。Open Questions でも
  採用要否は人間レビューに委ねる対象と明記されている。
- したがって、本実装では NFR 2.1（指定メジャー.マイナー以上の任意パッチを許容）を満たす形で
  `go 1.25` のみとし、Requirement 3 はあくまで境界値要件として扱う。

採用要否の最終確定は人間レビューに委ねる（後方互換的にいつでも `toolchain go1.25.x` を追記可能）。

## CI への影響確認結果

- `.github/workflows/ci.yml` の `backend` ジョブは `actions/setup-go@v5` の
  `go-version-file: go.mod` を参照している（18 行目）。
- `go-version-file` は `go.mod` の `go` ディレクティブをそのまま読み取り、指定された
  メジャー.マイナー（`1.25`）系列の Go をセットアップする。`go 1.25` 表記でも従来どおり
  解決できるため、ワークフロー定義の変更は不要。
- NFR 1.2（変更前後で同一ジョブ構成を維持）・Requirement 2.3 / 2.4（CI が
  `go-version-file: go.mod` でツールチェインを解決し成功する／パッチ差異でも成功する）を
  満たすため、`ci.yml` は変更しない。

## 検証コマンドの実行結果

ローカル環境の PATH 上の `go` は go1.22.2（ベース）で、`GOTOOLCHAIN=auto` のもとでは
キャッシュ済み go1.25.x へ昇格して動作する。本実装の検証は、実際の go1.25.1 ツールチェイン
（`golang.org/toolchain@v0.0.1-go1.25.1`）をベースとして実行した。

| 検証 | コマンド | 結果 |
|---|---|---|
| build | `go build ./...` | 成功（BUILD OK） |
| test | `go test ./...` | 全パッケージ成功（全 `ok`、failure なし） |
| vet | `go vet ./...` | 成功（VET OK） |
| mod verify | `go mod verify` | `all modules verified` |
| mod tidy 差分 | `go mod tidy` 後の diff | `go.mod` / `go.sum` ともに差分なし |
| gofmt | `gofmt -l .` | 既存 `.go` ファイルに pre-existing な差分あり（後述・スコープ外） |

### gofmt に関する補足

`gofmt -l .` は `internal/auth/google_oauth.go` 等いくつかの既存 `.go` ファイルを
列挙するが、これらは本 Issue 着手前から存在する pre-existing な整形差分であり、本変更
（`go.mod` の 1 行のみ）とは無関係。`go.mod` は gofmt の対象外（go ツールが整形を管理）で
あり、本変更によって新たな整形差分は発生していない。既存 `.go` ファイルの整形は本 Issue の
スコープ外のため修正しない。

### ローカル環境に関する注記

PATH 上のデフォルト `go` が go1.22.2 であるため、`GOTOOLCHAIN=auto` のまま `go 1.25` 指定の
`go.mod` に対して `go build` を実行すると、auto リゾルバが存在しないリリース名 `go1.25`
（正式リリースは `go1.25.0` / `go1.25.1` 等）をダウンロードしようとし、サンドボックス
（ネットワーク遮断）で失敗する。これはローカル環境固有の事象であり、`go 1.25` 表記自体の
不備ではない。実際の go1.25.x ツールチェインをベースに実行すると build / test / vet すべて
成功することを確認済み（上表参照）。CI（`actions/setup-go` がネットワーク到達可能で
go1.25 系列を正規取得する環境）では問題は発生しない。
