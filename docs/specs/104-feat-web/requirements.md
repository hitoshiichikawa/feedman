# Requirements Document

## Introduction

Feedman の記事一覧（右ペインのアイテム一覧）は現在、各記事についてタイトルと公開日時のみを
表示している。記事の概要（summary）が一覧に出ないため、ユーザーは内容を把握するために
各記事を開く操作を繰り返す必要があり、一覧性が低い。記事の概要は既にバックエンドの記事
ドメインモデル・リポジトリ取得結果（一覧クエリ）に格納されており、サニタイズ済みである。
本機能は、この既存の概要データを記事一覧 API レスポンスに含め、フロントエンドの各記事行で
タイトルの下に表示し、合わせて公開日時の表示位置をタイトルの右側へ移すことで、一覧上での
内容把握を容易にすることを目的とする。

## Requirements

### Requirement 1: 記事一覧 API レスポンスへの概要の追加

**Objective:** As a 記事一覧を閲覧するユーザー, I want 記事一覧の取得結果に各記事の概要が含まれること, so that 一覧画面で内容を把握できる

#### Acceptance Criteria

1. When クライアントが記事一覧を取得したとき, the 記事一覧 API shall 各記事のレスポンスにサニタイズ済みの概要テキストを含める
2. The 記事一覧 API shall 概要を記事詳細レスポンスと同一の文字列値で返す（一覧と詳細で概要内容を一致させる）
3. If 記事の概要が空であるとき, the 記事一覧 API shall 概要フィールドを空文字列として返す（フィールド自体は省略しない）
4. The 記事一覧 API shall 現状返している既存フィールド（タイトル・リンク・公開日時・推定フラグ・既読状態・スター状態・はてなブックマーク数）を変更せず保持する

### Requirement 2: 記事行への概要表示

**Objective:** As a 記事一覧を閲覧するユーザー, I want 各記事行のタイトル直下に概要が表示されること, so that 記事を開かずに内容の見当をつけられる

#### Acceptance Criteria

1. When 記事一覧が表示されるとき, the 記事一覧画面 shall 各記事行のタイトルの下に概要テキストを表示する
2. The 記事一覧画面 shall 概要テキストをタイトルより小さいフォントサイズで表示する
3. The 記事一覧画面 shall 概要テキストをタイトルより視覚的コントラストが低い（薄い）配色で表示し、タイトルと視覚的に区別できるようにする
4. If 記事の概要が空であるとき, the 記事一覧画面 shall 概要の表示領域を描画せず、当該記事行の高さを概要なし相当に保つ
5. While 記事が既読状態であるとき, the 記事一覧画面 shall 既存の既読表現（行全体の不透明度低下）を概要テキストにも一貫して適用する

### Requirement 3: 長い概要の省略表示

**Objective:** As a 記事一覧を閲覧するユーザー, I want 長い概要が一定行数で省略表示されること, so that 一覧の各行が縦に伸びすぎず一覧性が保たれる

#### Acceptance Criteria

1. When 概要が表示行数の上限を超える長さであるとき, the 記事一覧画面 shall 概要テキストを 1 行以上 2 行以内で打ち切り、省略表現を付与する
2. The 記事一覧画面 shall 1 記事行あたりの概要表示領域を最大 2 行に制限し、それ以上に行を増やさない
3. When 概要が表示上限以下の短い長さであるとき, the 記事一覧画面 shall 概要テキスト全体を打ち切りなしで表示する

### Requirement 4: タイムスタンプのタイトル右側配置

**Objective:** As a 記事一覧を閲覧するユーザー, I want 公開日時がタイトルの右側に配置されること, so that 概要を表示してもタイトル行で日時を確認できる

#### Acceptance Criteria

1. When 記事行が表示されるとき, the 記事一覧画面 shall 公開日時をタイトルと同一行の右側に配置する
2. While 公開日時が推定日時であるとき, the 記事一覧画面 shall 既存の推定フラグ表示を公開日時に隣接して維持する
3. When 表示幅が狭くタイトルと公開日時が同一行に収まらないとき, the 記事一覧画面 shall タイトル・公開日時・概要のいずれの要素も重なり・はみ出しによる表示崩れを起こさずに配置する
4. The 記事一覧画面 shall 公開日時をタイトルより縮小されないよう、日時テキストが省略・切り詰めされても日時の意味が判読できる範囲に保つ

### Requirement 5: 一覧件数・既存挙動の維持

**Objective:** As a 記事一覧を閲覧するユーザー, I want 概要追加後も一覧の表示件数やスクロール挙動が維持されること, so that 一覧性の改善が既存の閲覧体験を損なわない

#### Acceptance Criteria

1. The 記事一覧画面 shall 1 回の取得で表示する記事件数を現状設計（既存のページサイズ）から減少させない
2. While ユーザーが一覧をスクロールするとき, the 記事一覧画面 shall 既存の追加読み込み（無限スクロール）挙動を維持する
3. The 記事一覧画面 shall 記事行クリックによる記事選択・展開の既存挙動を維持する

## Non-Functional Requirements

### NFR 1: 互換性

1. The 記事一覧 API shall 既存のレスポンス項目名・型を変更せず、概要フィールドの追加のみで後方互換を保つ
2. While 概要フィールドを認識しない既存クライアントが利用するとき, the 記事一覧 API shall 追加フィールドの存在によって既存クライアントの記事一覧表示を破綻させない

### NFR 2: セキュリティ

1. The 記事一覧画面 shall バックエンドでサニタイズ済みの概要をそのまま表示し、フロントエンド側で HTML を生 DOM へ未サニタイズのまま注入しない

### NFR 3: 一覧性（ユーザー可視の表示量）

1. The 記事一覧画面 shall 概要を 2 行以内に制限することで、概要追加前と比較して 1 画面あたりに視認できる記事件数の極端な減少（半減以下）を防ぐ

## Out of Scope

- 記事本文（content）のローディング処理および記事詳細画面のレイアウト変更
- 概要から HTML タグを除去する独自パーサーの新規実装（バックエンドでサニタイズ済みの概要を表示するのみ）
- 概要のキーワードハイライト・全文検索・翻訳などの付加機能
- 概要表示の有無や表示行数をユーザーが切り替える設定 UI
- 概要の生成・更新ロジック（フィード取得・サニタイズ処理）の変更

## Open Questions

- 概要の省略行数（Requirement 3）について、Issue では「1〜2 行」とあるが確定行数（1 行固定か 2 行か）が明記されていない。本要件では「最大 2 行」として整理したが、デザイン上の確定値があれば指定が必要。
- タイトルと公開日時が同一行に収まらない狭幅時（Requirement 4.3）の優先表示（タイトルを優先して日時を折り返す／省略する等）の確定挙動は明記されていない。本要件では「表示崩れを起こさない」までを必須化し、具体的な折り返し方針は design 側の判断に委ねる。
