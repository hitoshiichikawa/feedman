# Requirements Document

## Introduction

Feedman の既存レート制限ミドルウェアは、リクエストコンテキストから userID を引いてユーザー単位で制限する設計のため、セッションを経由しない未認証エンドポイント（`/auth/google/login`・`/auth/google/callback`・`/health`）はレート制限の対象外になっている。この空白は、OAuth フローへの大量リクエスト（OAuth フラッディング）や `/health` 経由の継続的な過負荷の入口になりうる。本要件では、これら未認証エンドポイントに対して同一クライアント（IP）単位のレート制限を適用し、閾値超過時に `429 Too Many Requests` を返すことを定義する。同時に、既存の認証済みエンドポイントにおける userID ベースのレート制限挙動は一切変更せず後方互換を維持する。

## Requirements

### Requirement 1: 未認証エンドポイントへの IP 単位レート制限

**Objective:** As a サービス運用者, I want 未認証エンドポイントを同一 IP 単位でレート制限したい, so that OAuth フラッディングや /health 経由の過負荷からサービスを保護できる

#### Acceptance Criteria

1. When 同一クライアント IP から `/auth/google/login` への単位時間あたりリクエスト数が閾値を超過したとき, the 認証エンドポイントレート制限 shall 以降のリクエストに対して HTTP 429 Too Many Requests を返す
2. When 同一クライアント IP から `/auth/google/callback` への単位時間あたりリクエスト数が閾値を超過したとき, the 認証エンドポイントレート制限 shall 以降のリクエストに対して HTTP 429 Too Many Requests を返す
3. When 同一クライアント IP から `/health` への単位時間あたりリクエスト数が閾値を超過したとき, the 認証エンドポイントレート制限 shall 以降のリクエストに対して HTTP 429 Too Many Requests を返す
4. While 同一クライアント IP からのリクエスト数が閾値以内であるとき, the 認証エンドポイントレート制限 shall リクエストを後続ハンドラへ通常どおり通過させる
5. When 異なるクライアント IP から同時にリクエストが到達したとき, the 認証エンドポイントレート制限 shall 各 IP のリクエスト数を独立にカウントする
6. When 閾値超過により拒否したとき, the 認証エンドポイントレート制限 shall 再試行可能になるまでの待機時間をレスポンスヘッダーで通知する

### Requirement 2: レート制限閾値の設定可能性

**Objective:** As a サービス運用者, I want IP 単位レート制限の閾値を環境ごとに調整したい, so that デプロイ環境のトラフィック特性に合わせて過剰遮断と過小防御を回避できる

#### Acceptance Criteria

1. The 認証エンドポイントレート制限 shall IP 単位の閾値の既定値として 30 リクエスト毎分毎 IP を適用する
2. Where 運用者が IP 単位の閾値を設定値として指定したとき, the 認証エンドポイントレート制限 shall 既定値ではなく指定された閾値を適用する
3. If 指定された閾値の設定値が不正であるとき, the 認証エンドポイントレート制限 shall 既定値を採用して起動を継続する

### Requirement 3: クライアント IP 判定方針

**Objective:** As a セキュリティ担当者, I want クライアント IP の判定方針が明示的に定義されていてほしい, so that リバースプロキシ配下でのなりすましヘッダーによる制限回避や巻き添え遮断を防げる

#### Acceptance Criteria

1. The 認証エンドポイントレート制限 shall クライアント IP を接続元アドレスから判定する
2. The 認証エンドポイントレート制限 shall クライアント IP 判定において X-Forwarded-For ヘッダーを信頼しない
3. If 接続元アドレスからクライアント IP を判定できないとき, the 認証エンドポイントレート制限 shall 当該リクエストを安全側で扱い無制限通過を許さない

### Requirement 4: 認証済みエンドポイントの後方互換

**Objective:** As a 既存利用者, I want 認証済みエンドポイントのレート制限挙動が変わらないでほしい, so that 本変更導入後も従来どおりの利用体験が維持される

#### Acceptance Criteria

1. The 認証済みエンドポイントレート制限 shall 本変更導入前と同一の userID 単位の制限挙動を維持する
2. When 認証済みエンドポイントへリクエストが到達したとき, the システム shall IP 単位レート制限ではなく従来の userID 単位レート制限を適用する
3. While 同一ユーザーが複数の IP からアクセスしているとき, the 認証済みエンドポイントレート制限 shall 従来どおり IP に依存せず userID 単位でカウントする

## Non-Functional Requirements

### NFR 1: 可観測性

1. When IP 単位レート制限により拒否が発生したとき, the 認証エンドポイントレート制限 shall 拒否事象を運用ログとして記録する
2. The 認証エンドポイントレート制限 shall 拒否ログに個人を特定しうるセッション情報やトークンを含めない

### NFR 2: 互換性

1. While IP 単位レート制限の設定値が未指定であるとき, the システム shall 既定値で起動し設定追加なしに本機能を有効化する
2. The システム shall 未認証エンドポイント以外の既存ルーティングおよびミドルウェア順序を変更しない

### NFR 3: 資源管理

1. The 認証エンドポイントレート制限 shall IP 単位の内部状態を一定期間アクセスのない IP について解放し、無制限なメモリ増加を防ぐ

## Out of Scope

- 認証済みエンドポイント（`/api/*`）のレート制限ロジックの根本再設計
- 分散環境で複数インスタンス間にレート制限状態を共有する外部ストア（Redis 等）の導入
- X-Forwarded-For を信頼する運用への切替や、信頼プロキシ階層数の動的設定
- IP 単位制限のユーザー別・エンドポイント別の細粒度ポリシー化（本要件は未認証エンドポイント群に一律の IP 単位制限を適用する）
