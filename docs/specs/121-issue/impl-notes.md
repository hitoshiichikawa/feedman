### Task 3

- 採用方針: `ItemRepository` interface に `ListNewAcrossFeeds(ctx, userID, sinceTime, cursorPublishedAt, cursorItemID, limit) ([]CrossFeedItem, error)` を追加し、`postgres_item_repo.go` で `items × subscriptions × feeds × item_states (LEFT)` を 1 クエリで JOIN する実装を追加。cursor 有無で SQL を分岐し、cursor あり時のみ `(i.published_at, i.id) < ($3, $4::uuid)` のタプル比較を WHERE に含める（plain text の文字列比較を避けるため `::uuid` キャストを付与し、`SearchByUserAndKeyword` の既存パターンと整合）。新規 row 型 `CrossFeedItem` は `model.ItemWithState` を embed し `FeedTitle` / `FaviconData` / `FaviconMime` を併記する形で `interfaces.go` に追加した。
- 重要な判断: (1) cursor の有効性判定は `!cursorPublishedAt.IsZero() && cursorItemID != ""` の AND 条件とし、片方のみゼロ値で渡された場合は cursor なし扱い（先頭から取得）に倒すことで呼び出し側のバリデーション漏れを安全側に倒した。(2) integration test は `setupListDueTestDB` の既存ヘルパを流用するが、ヘルパ `insertCrossFeedTestItem` / `insertCrossFeedTestItemState` / `updateCrossFeedFeedFavicon` / `sortDescending` は同パッケージ内の他テスト群（`insertTestItem` / `insertStarredTestItem` 等）との命名衝突を避けるため、テストの目的を表す接頭辞 `CrossFeed` を付与した独立命名で定義した（task 2 で確立済の命名分離パターンに準拠）。テストは独立ファイル `postgres_item_repo_cross_feed_test.go` に分離し、既存 `postgres_item_repo_test.go` には触れていない。(3) `ItemRepository` interface への新メソッド追加に伴い `internal/item/service_test.go` の `mockItemRepoForService` と `internal/item/upsert_test.go` の `mockItemRepo` の interface 適合が失われたため、両モックに `ListNewAcrossFeeds` の最小スタブ（常に nil を返す）を追加した。これは task 3 の boundary（ItemRepository）の interface 拡張に追随する不可避な修正であり、`item` ドメインの挙動は変更していない。
- 残存課題: なし。task 1, 2 で言及されていた `postgres_item_repo_starred_test.go` での `insertTestItem` 重複事象は本 task 着手時点では **解消済み**（`grep -rn "func insertTestItem"` で `postgres_item_repo_search_test.go` の 1 箇所のみヒット）であり、`go vet ./...` / `go build ./...` / `go test -count=1 ./internal/repository/... ./internal/item/...` がいずれもクリーンに通る状態に復旧している。DB 結合テスト本体は `TEST_DATABASE_URL` が未設定の環境（CI / 本作業環境）ではスキップされるため、`go test -short` ではテスト本体が起動しない点は既存パターンと同じ。

### Task 4

- 採用方針: `internal/crossfeed/service.go` に `Service` / `NewItemsResult` / `CrossFeedItemSummary` と `ListNewItems` / `TouchLastSeen` を新規実装。sinceTime 決定は `resolveSinceTime` ヘルパに切り出して 3 段優先順位（overrideSince → stored lastSeen → now-24h fallback）を直線的に表現。cursor parse は `strings.LastIndex(s, ":")` で末尾の `:` を分割位置とし、RFC3339Nano が `:` を含む点を吸収（`item.parseItemCursor` と異なり複合カーソル対応のため独自実装）。favicon の data URL 構築は `internal/subscription/service.go` の `fmt.Sprintf("data:%s;base64,%s", mime, base64.StdEncoding.EncodeToString(data))` 方式と完全に整合。
- 重要な判断: (1) `CrossFeedItemSummary` は design.md の参考 Go 定義では `item.ItemSummary` を embed しつつ outer に `FeedID` を再宣言していたが、`item.ItemSummary` 自体が既に `FeedID` を持つため二重宣言となりコンパイル不可。embed 側のみで `FeedID` を運ぶ形に修正し、outer は `FeedTitle` / `FeedFaviconURL` のみとした（design.md の意図は満たしつつ Go 言語仕様に適合）。本判断は impl-notes 領分のため design.md には反映していない。(2) `nowFn func() time.Time` を Service の内部 hook として導入し、テストで固定時刻（fallback 24h 検証 / TouchLastSeen の now 引数検証）を注入可能にした。production 経路では nil のまま `time.Now()` が使われ既存挙動と等価。(3) `parseCrossFeedCursor` は `time.Parse(RFC3339Nano)` 失敗時に `time.Parse(RFC3339)` も試す fallback を入れて呼び出し側の正規化負担を緩めた（既存 `item.parseItemCursor` と同方針）。不正形式は `model.NewInvalidFilterError("invalid cursor: " + cursorStr)` で既存エラーコード `INVALID_FILTER` を再利用（新規コード追加なし）。(4) Repository モックは `internal/item/service_test.go` の `mockItemRepoForService` パターンに倣い、本テストで使う `ListNewAcrossFeeds` のみ関数差し替え可能にし、他 ItemRepository メソッドは no-op スタブで interface 適合のみ満たす形にした。
- 残存課題: なし。後続 task 5（CrossFeedHandler）が本 Service の `ListNewItems` / `TouchLastSeen` を直接呼ぶことを想定したシグネチャに整合。design.md「Service Interface」の `time.Now()` ベース TouchLastSeen と一致。

### Task 5

- 採用方針: `internal/handler/crossfeed_handler.go` に `CrossFeedServiceInterface` / `CrossFeedHandler` / `crossFeedItemResponse` / `crossFeedListResult` を新規実装。GET /api/items/cross-feed と PUT /api/users/me/cross-feed-last-seen を分離（PM 確認事項 / リトライ冪等性のため）。`CrossFeedServiceAdapter` は既存 `ItemServiceAdapterFromDomain` / `ItemSearchServiceAdapter` と同パターンで domain → handler DTO 変換。`internal/app/app.go` の `runServe` に DI 配線追加（`userCrossFeedViewRepo` → `crossfeed.NewService(itemRepo, userCrossFeedViewRepo)` → `handler.NewCrossFeedServiceAdapter` → `deps.CrossFeedService`）。
- 重要な判断: (1) `crossFeedItemResponse.FeedFaviconURL` は `*string` で **json タグから `omitempty` を外した**（`feed_favicon_url` フィールド自体は常に存在、未設定時は明示的に `null`）。これは Req 3.3 のクライアント側 fallback 判定（favicon 未設定なら `Rss` 代替アイコン表示）が「null vs 文字列」で判別する設計で、`omitempty` だとフィールド自体が消えクライアントが「未設定」を検知できなくなるため。既存 `itemSearchHitResponse.FaviconURL` は `omitempty` 付きだが、設計者の意図（design.md「レスポンス DTO」節）が常時 nullable フィールドを意図しているため本機能は別方針を採用した。(2) router 登録は `/api/items/cross-feed` を `/api/items/{id}` より **前**に登録し、chi v5 トライ木の静的セグメント優先動作に明示的に依存する（既存 `/api/feeds/starred/items` と同じ保護パターン）。(3) `RouterDeps.CrossFeedService` 配線が無い deps（既存テストの大半）でも router 構築が失敗しないよう、`crossFeedHandler` を nil ガードで作成し当該ルート登録もスキップする（後方互換、router_full_test 等の既存テストが破壊されない）。(4) handler テストは個別 handler 経由（13 件）+ router 経由（2 件: 正常ルーティング / 401）の 2 層で構成。router 経由テストでは既存 `mockSessionFinderForRouter` / `mockFeedService` 等を流用する `minimalRouterDepsForCrossFeed` ヘルパを切り出し、CrossFeedService 以外の handler 群は他ルートに到達しない前提で zero-value mock を渡す（NewRouter 構築時の参照のみ満たせばよい）。(5) `since` query parameter のパースは `time.Parse(time.RFC3339, v)` を使用（RFC3339Nano も受け付ける後方互換挙動）。失敗時は `INVALID_REQUEST` カスタム APIError（既存 `model.NewInvalidFilterError` は cursor 専用と整理）。
- 残存課題: なし。後続 task 6〜9（Frontend）が本 handler の API 契約（`feed_favicon_url` が `null` 可能、`since_time` 常時返却、`limit` 既定 50 / 上限 200）に依存することを前提に DTO を設計済。

### Task 6

- 採用方針: `FeedFavicon` を `web/src/components/feed-favicon.tsx` に named export として挙動不変で抽出し、`feed-list.tsx` の内部 private function 定義を削除して新パスから import。`AppStateContext` に `viewMode: 'none' | 'feed' | 'cross-feed'`（初期 `'none'`）と `crossFeedSessionSince: string | null`（初期 `null`、Req 4.7）を **追加**フィールドとして導入。`SELECT_ALL_NEW_ITEMS` action（viewMode='cross-feed' + selectedFeedId/expandedItemId reset + filter='all'、`crossFeedSessionSince` 保持）と `SET_CROSS_FEED_SESSION_SINCE` action（payload `sinceTime` で固定）を新設。既存 `SELECT_FEED` は viewMode='feed' に倒すロジックを追加（`crossFeedSessionSince` は保持、Req 4.7）。
- 重要な判断: (1) design.md / tasks.md は `AppState.viewMode: 'none' | 'feed' | 'cross-feed'` の単一 discriminator を前提としていたが、現行コードには #161 等で追加済の `selectedView: 'feed' | 'starred'`（starred 横断一覧用）が既に存在。両者を **並列の discriminator** として共存させる方針を採用した（既存 starred 機能との非リグレッション維持を優先）。`SELECT_ALL_NEW_ITEMS` は `selectedView='feed'` に倒して starred モードと排他にし、`SELECT_FEED` は両方の discriminator を整合させる。AppShell の分岐順序（task 9）では `isSearching > selectedView==='starred' > viewMode==='cross-feed' > 既存 ItemList` の優先で配線される想定。本判断は impl-notes 領分のため design.md / tasks.md は書き換えていない（後述「確認事項」に Reviewer 確認向けに記録）。(2) FeedFavicon は元コードを **完全に挙動不変**で切り出し、`useState` / props 型 / className / data-testid / `<img onError>` → fallback ロジックも変更なし。`lucide-react` の `Rss` import を新ファイル側に移し、`feed-list.tsx` 側からは `Rss` の import を削除した（dead import 除去）。(3) reducer test は既存「初期状態」テストに `viewMode='none'` / `crossFeedSessionSince=null` のアサーションを追加し、新規 4 テストで (a) SELECT_ALL_NEW_ITEMS の遷移、(b) SELECT_FEED で viewMode='feed' になること、(c) 横断 ⇄ 個別の往復で `crossFeedSessionSince` 保持、(d) SET_CROSS_FEED_SESSION_SINCE で固定、を網羅した（_Requirements: 1.2, 1.3, 4.7_）。FeedFavicon test は (a) faviconURL あり → `<img>`、(b) null → fallback、(c) 空文字 → fallback（境界値）、(d) onError 発火 → fallback 切替、の 4 ケース。
- 残存課題: (a) 後続 task 7 で FeedList props に `viewMode` / `onSelectAllNewItems` 追加が必要。本 task では `viewMode` を AppState に追加したのみで FeedList の signature は変更していない。(b) 後続 task 9 で AppShell の右ペイン分岐に `viewMode==='cross-feed'` 用の `<CrossFeedItemList />` 描画を追加する際、既存 `selectedView==='starred'` / `isSearching` との優先順位を明示的に決める必要がある（上記重要な判断 (1) の通り、本 impl-notes で示唆した順序を尊重するか task 9 着手時点で再確認）。

## 確認事項

- task 1 で migration timestamp が design.md 指定（`20260528120000`）と既存 migration `20260528120000_add_item_search_indexes` で衝突したため、`20260528130000` に micro-adjust した。後続 task の本文中に同 timestamp を参照する箇所は無いため影響なし（design.md は書き換えていない）。
- `go vet ./...` 全体実行時に `internal/repository/postgres_item_repo_starred_test.go` で `insertTestItem redeclared in this block` の既存事象が出力されたが、本 task で触れていないファイル群であり、本 task の変更とは無関係（変更パッケージ `internal/model` / `internal/database` 単体での vet は pass）。後続 task または別 Issue での対処を要する可能性がある旨を Reviewer / PM に共有する。
- task 6: design.md / tasks.md は `AppState.viewMode` を単一の view discriminator として導入することを前提に書かれているが、当該 spec 起草以降に `selectedView: 'feed' | 'starred'`（横断スター記事一覧、Issue #161 等）が main に merge 済み。本 task では既存 `selectedView` を削除せず `viewMode` を並列の discriminator として導入し、`SELECT_ALL_NEW_ITEMS` 時に `selectedView` を `'feed'` に倒すことで starred モードと排他にする方針を採用した（既存 starred 機能の非リグレッション維持優先）。本判断は spec 範囲を逸脱しないが、Reviewer / PM は (a) `selectedView` を `'feed' | 'starred' | 'cross-feed'` に拡張して `viewMode` を導入しない案、(b) 本 task 採用の並列導入案、のどちらが望ましいかを確認したい場合は task 9 着手前に方針を共有してほしい。
