# Requirements Document

## Introduction

worker のフェッチスケジューラは、定期的に「取得期限を迎えたフィード（due feed）」を取得し、バックグラウンドで RSS/Atom を取り込むコア機能を担う。しかし現状はフェッチ対象を取得するクエリが毎サイクル即座に失敗しており、フィードのバックグラウンド取得が一切機能していない（登録しても記事が増えない、空 DB でもクエリ自体が不正で失敗する）。原因はフェッチ対象取得処理で「重複排除」と「行ロック（排他取得）」の機構が PostgreSQL 上で同時に成立しない組み合わせになっていることにある。本要件は、worker のフェッチサイクルが正常に復旧し、購読者が存在する due feed を重複なく排他的に取得できる状態へ戻すことを目的とする。実装手法（具体的な SQL の書き換え方）は design の領分とし、本書では観測可能な振る舞いのみを定義する。

## Requirements

### Requirement 1: フェッチ対象取得の正常復旧

**Objective:** As a worker 運用者, I want フェッチ対象フィードの取得処理がエラーなく完了すること, so that フィードのバックグラウンド取得サイクルが復旧し記事が定期的に増える

#### Acceptance Criteria

1. When フェッチスケジューラがフェッチ対象フィードの取得を実行したとき, the フェッチ対象取得処理 shall データベースエラーを発生させずに結果を返す
2. When 取得期限を迎えた購読済みフィードが 1 件以上存在する状態でフェッチサイクルが実行されたとき, the フェッチスケジューラ shall 対象フィードを取得してフェッチ処理へ引き渡す
3. While 購読フィードが 0 件の空のデータ状態であるとき, the フェッチ対象取得処理 shall エラーを発生させず空の結果を返す

### Requirement 2: フェッチ対象の選別条件

**Objective:** As a worker 運用者, I want 取得対象が「取得期限を迎え」「アクティブで」「購読者が存在する」フィードに正しく限定されること, so that 取得すべきでないフィードに対する無駄なフェッチや取りこぼしが発生しない

#### Acceptance Criteria

1. When 購読者が複数存在する 1 つのフィードがフェッチ対象になったとき, the フェッチ対象取得処理 shall そのフィードを 1 回だけ結果に含める
2. When フェッチ対象取得処理が実行されたとき, the フェッチ対象取得処理 shall 購読者が 1 人以上存在するフィードのみを結果に含める
3. If あるフィードの購読者が 0 人であるとき, the フェッチ対象取得処理 shall そのフィードを結果から除外する
4. When フェッチ対象取得処理が実行されたとき, the フェッチ対象取得処理 shall 取得期限（next_fetch_at）が現在時刻以下のフィードのみを結果に含める
5. If あるフィードの取得期限（next_fetch_at）が現在時刻より後であるとき, the フェッチ対象取得処理 shall そのフィードを結果から除外する
6. When フェッチ対象取得処理が実行されたとき, the フェッチ対象取得処理 shall フェッチ状態（fetch_status）が active のフィードのみを結果に含める
7. If あるフィードのフェッチ状態（fetch_status）が active 以外であるとき, the フェッチ対象取得処理 shall そのフィードを結果から除外する

### Requirement 3: 排他取得の維持

**Objective:** As a worker 運用者, I want フェッチ対象フィードが行単位で排他的に取得されロック済みの行はスキップされること, so that 複数 worker / 並列サイクルが同一フィードを二重にフェッチしない

#### Acceptance Criteria

1. When フェッチ対象取得処理が実行されたとき, the フェッチ対象取得処理 shall 取得したフィード行に対して取得側でロックを獲得する
2. While あるフィード行が別のトランザクションによってロックされているとき, the フェッチ対象取得処理 shall その行をスキップして他の取得可能なフィードを返す

### Requirement 4: 回帰テストによる検証

**Objective:** As a 開発者, I want 本不具合の再発を検知する回帰テストが用意されること, so that 将来の変更でフェッチ対象取得が再び破壊された場合に CI で検出できる

#### Acceptance Criteria

1. The 回帰テスト shall 購読者が複数存在するフィードが結果に 1 回だけ含まれること（重複しないこと）を検証するケースを含む
2. The 回帰テスト shall 取得期限を迎えていないフィード、および active 以外のフィードが結果から除外されることを検証する境界・異常系ケースを含む
3. The 回帰テスト shall 購読者が 0 人のフィードが結果から除外されることを検証するケースを含む
4. The 回帰テスト shall 購読フィードが存在しない空のデータ状態でエラーなく空の結果が返ることを検証するケースを含む

## Non-Functional Requirements

### NFR 1: 後方互換性

1. The フェッチ対象取得処理 shall 修正前と同一の呼び出しインターフェース（引数と返却される値の意味）を維持し、呼び出し側であるフェッチスケジューラの変更を不要とする

### NFR 2: 可観測性

1. If フェッチ対象取得処理がデータベースエラーで失敗したとき, the フェッチスケジューラ shall そのエラーをログに記録する

## Out of Scope

- フェッチ間隔（next_fetch_at の更新ロジック）やバックオフ戦略の変更
- フェッチ並列数・スケジューラの周期など、取得対象選別以外のスケジューラ挙動の変更
- 購読・フィード登録 API 側の挙動変更
- 取得対象選別条件そのものの仕様変更（active 判定基準・due 判定基準は現行仕様を維持する）
- 具体的な SQL 文・テーブル定義・インデックス設計（design.md の領分）
