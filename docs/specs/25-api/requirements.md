# Requirements Document

## Introduction

Feedman の Web フロントエンドでは、各カスタムフックおよび一部コンポーネントが実行（≒コンポーネントのレンダリング）のたびに `createApiClient()` を呼び出して API クライアントを新規生成している。API クライアントはステートレス（リクエスト関数をラップするだけでインスタンス固有の状態を持たない）であり、再生成する必要がない。本機能は API クライアントをモジュールレベルのシングルトン（共有インスタンス）に集約し、レンダリングごとの不要な生成を排除する技術債返済（優先度 Low）である。フックの公開インターフェース（戻り値・引数・呼び出し方）は変更せず、利用側の画面挙動は従来と完全に同一であることを前提とする。`createApiClient()` 自体の内部実装（認証・ベース URL 解決等）の変更は本機能のスコープに含まない（ベース URL 解決の同一オリジン化は #41 / PR #62 で既に merge 済み）。

## Requirements

### Requirement 1: API クライアント生成回数の削減

**Objective:** As a Feedman の保守担当者, I want API クライアントがモジュール初期化時に一度だけ生成され各消費者が共有インスタンスを参照すること, so that レンダリングごとの不要なインスタンス生成を排除し技術債を返済できる

#### Acceptance Criteria

1. The Web フロントエンドの API クライアント参照機構 shall モジュール初期化時に一度だけ API クライアントを生成し、以降は同一の共有インスタンスを返す
2. When API クライアントを参照するフックまたはコンポーネントが複数回実行された場合, the Web フロントエンドの API クライアント参照機構 shall 追加の API クライアントインスタンスを生成しない
3. When 複数の異なるフックまたはコンポーネントが API クライアントを参照する場合, the Web フロントエンドの API クライアント参照機構 shall それらすべてに同一の API クライアントインスタンスを参照させる

### Requirement 2: 利用側挙動の不変性（後方互換）

**Objective:** As a Feedman のユーザー, I want フィード一覧・アイテム一覧・認証・アイテム状態・購読操作の画面挙動が従来と変わらないこと, so that 内部リファクタによって既存機能が壊れない

#### Acceptance Criteria

1. While API クライアントが共有インスタンス化された状態で, the Web フロントエンド shall フィード一覧の取得・表示を従来と同一の結果で行う
2. While API クライアントが共有インスタンス化された状態で, the Web フロントエンド shall アイテム一覧の取得・表示を従来と同一の結果で行う
3. While API クライアントが共有インスタンス化された状態で, the Web フロントエンド shall 認証状態の取得および認証操作を従来と同一の結果で行う
4. While API クライアントが共有インスタンス化された状態で, the Web フロントエンド shall アイテム状態（既読・未読等）の更新を従来と同一の結果で行う
5. While API クライアントが共有インスタンス化された状態で, the Web フロントエンド shall 購読の登録・解除・設定更新を従来と同一の結果で行う
6. The 各フックの公開インターフェース shall 戻り値・引数・呼び出し方を変更せず、利用側コンポーネントの修正を不要とする

### Requirement 3: SSR / クライアント境界とテスト容易性の維持

**Objective:** As a Feedman の保守担当者, I want SSR / クライアント境界の前提とテスト時のモック差し替え可能性が維持されること, so that 既存のランタイム前提とテストスイートを壊さずにリファクタを完了できる

#### Acceptance Criteria

1. The Web フロントエンド shall API クライアント共有インスタンス化の前後でクライアント側実行を前提とする境界（`"use client"` 前提）を変更しない
2. Where テストが API 通信をモック差し替えして検証する場合, the API クライアント参照機構 shall モック差し替えが可能な形を維持する
3. The Web フロントエンドのテストスイート shall 本変更の前後で既存テストを破壊せず、変更後も全て pass する

## Non-Functional Requirements

### NFR 1: 後方互換（公開インターフェースの不変性）

1. The 各フックの公開インターフェース shall 破壊的変更（戻り値の型・引数・関数シグネチャの変更）を含まない

### NFR 2: 観測可能な性能特性

1. While 同一フックが複数のコンポーネントで使用されている状態で, the Web フロントエンド shall API クライアントインスタンスを 1 個（共有インスタンス）に保ち、コンポーネント数に比例した追加生成を行わない

## Out of Scope

- `createApiClient()` 自体の内部実装の変更（認証ヘッダー付与・ベース URL 解決等）。なおベース URL 解決の同一オリジン化は #41 / PR #62 で既に merge 済みであり、本機能では触れない
- API クライアントへの新機能追加（リトライ・キャッシュ・タイムアウト制御等）
- TanStack Query のキャッシュ戦略・queryKey 設計の変更
- バックエンド（`api` / `worker`）側の変更

## Open Questions

- なし（Issue 本文・Triage コメント・既存実装・オーケストレーターの事前調査で範囲と期待挙動が確定している。消費者スコープについては、フック（`use-feeds` / `use-items` / `use-auth` / `use-item-state` / `use-subscriptions`）に加え、同一の毎レンダリング生成パターンを持つコンポーネント（`feed-register-dialog` / `withdraw-dialog`）も Requirement 1 / 2 の対象に含める方針を採用した。共有インスタンスの公開方法・テストのモック差し替え機構の具体は design / 実装の領分とする）
