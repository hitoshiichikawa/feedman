# Requirements Document

## Introduction

Feedman の既存スター機能は、特定フィードの記事一覧画面で「スター」フィルタを選択した場合に
限り、そのフィード内のスター付き記事のみを参照できる。複数フィードにまたがる「保存した記事」を
一覧する動線が存在しないため、ユーザーは過去にスターを付けた記事を再閲覧するために対象フィードを
記憶し個別に切り替える必要がある。本要件は、左ペインに「お気に入り」ナビゲーション項目を新設し、
これを選択した際に現ユーザーの全フィード横断スター付き記事を、既存の単一フィード記事一覧と同等の
ユーザー体験（公開日時降順・カーソルベースの追加読込・記事行クリック展開・スター解除即時反映）で
閲覧できるようにすることを目的とする。

なお、既存のスター更新エンドポイント、スター状態を保持するデータモデル、スター用の部分インデックス
（`idx_item_states_user_starred`）は既に存在するため、本要件はそれらの新規追加ではなく、既存資産を
前提に「フィード横断のスター記事一覧」を取得する経路と UI ナビゲーション項目の追加を扱う。
フォルダ分け・タグ付け等の整理機能は Issue 本文で明示的にスコープ外とされており、本要件にも
含めない。

## Goals

- 左ペインに「お気に入り」ナビゲーション項目を固定表示し、フィード一覧と並列に選択できる動線を提供する。
- 「お気に入り」選択時、右ペインに現ユーザーの全フィード横断スター付き記事一覧を公開日時降順で表示する。
- 既存のフィード別記事一覧（`GET /api/feeds/{feedID}/items`）と同形式のページネーション応答スキーマで
  「お気に入り」一覧を返却し、フロントエンドが同等のカーソル送り出し処理を再利用できるようにする。
- スター解除操作（既存挙動）の結果が、お気に入り一覧の再取得タイミングで反映される。

## Non-Goals

- お気に入り記事のフォルダ分け・タグ付け・並べ替えカスタマイズ（Issue 本文で明示的にスコープ外）。
- お気に入り以外のフィード横断ビュー（横断「全件」「未読」等）の追加。
- スター操作 API（既存）の挙動変更・新規エンドポイント追加。
- 既存のスター用部分インデックスの再設計・追加インデックスの導入判断（性能要件を満たす実装手段は
  `design.md` の領分）。
- お気に入り一覧における未読/全件サブフィルタの提供（既存フィード別一覧の `all`/`unread`/`starred`
  タブ相当の UI 切替は本要件のスコープ外）。

## Requirements

### Requirement 1: 左ペイン「お気に入り」ナビゲーション項目

**Objective:** As a Feedman ユーザー, I want 左ペインから「お気に入り」を選択できる固定項目があること, so that 個別フィードに切り替えることなく保存済み記事の一覧画面に直接遷移できる

#### Acceptance Criteria

1. The Web UI shall フィード一覧の左ペインに「お気に入り」という固定ナビゲーション項目を、登録済みフィード件数に依存せず常時 1 件表示する。
2. The 「お気に入り」項目 shall 現在選択中であることを視覚的に判別できる選択状態（アクティブ状態）を、既存のフィード行と同じ表示規約（背景色・フォントウェイト等）で提供する。
3. When ユーザーが「お気に入り」項目をクリックしたとき, the Web UI shall 右ペインの表示対象を「お気に入り（フィード横断スター記事一覧）」に切り替える。
4. When ユーザーが「お気に入り」項目を選択した状態から個別フィード行をクリックしたとき, the Web UI shall 「お気に入り」のアクティブ状態を解除し、選択されたフィードをアクティブにする。
5. While 「お気に入り」項目が選択されている間, the Web UI shall 既存のフィード一覧側で「いずれのフィード行もアクティブではない」状態を提示する。

### Requirement 2: 「お気に入り」選択時の右ペイン表示

**Objective:** As a Feedman ユーザー, I want お気に入り選択時に全フィード横断のスター記事一覧を時系列で確認できること, so that どのフィードに属するかを意識せず保存済み記事を再閲覧できる

#### Acceptance Criteria

1. While 左ペインで「お気に入り」項目が選択されている間, the Web UI shall 右ペインのコンテキストタイトルとして「お気に入り」を表示する。
2. While 「お気に入り」が選択されている間, the Web UI shall 現ユーザーが付与した全フィード横断のスター付き記事を、公開日時の降順（最新が先頭）で一覧表示する。
3. While 「お気に入り」一覧を表示している間, the Web UI shall 既存のフィード別記事一覧と同じ記事行レイアウト（タイトル・概要・公開日時・スターアイコン・既読の視覚的減衰）を再利用する。
4. While 「お気に入り」一覧を表示している間, the Web UI shall 各記事行にどのフィードに属する記事かを示す情報（フィードタイトル等のフィード識別情報）を併記して表示する。
5. When ユーザーが「お気に入り」一覧の末尾までスクロールしたとき, the Web UI shall 後続ページが存在する場合に追加読込を 1 回自動的に発行し、既存の無限スクロールと同等の操作感を維持する。
6. If 現ユーザーがスター付き記事を 1 件も持たないとき, the Web UI shall 「お気に入り」一覧の代わりに「記事がありません」相当の空状態メッセージを表示する。
7. If 「お気に入り」一覧の取得に失敗したとき, the Web UI shall エラー状態として「記事の読み込みに失敗しました」相当のメッセージを表示し、空一覧との区別を可能にする。
8. When ユーザーが「お気に入り」一覧の記事行をクリックしたとき, the Web UI shall 既存のフィード別一覧と同じ排他的展開（クリック行直下に記事詳細を表示・他展開は閉じる）を行う。

### Requirement 3: スター解除と一覧反映

**Objective:** As a Feedman ユーザー, I want お気に入りから解除した記事が一覧から消えること, so that お気に入り一覧が「現在スターが付いている記事」と一貫した状態を保つ

#### Acceptance Criteria

1. While 「お気に入り」一覧を表示している間, the Web UI shall 既存と同じスター切替操作（記事詳細からのスター解除）をそのまま利用可能とする。
2. When ユーザーが「お気に入り」一覧上の記事をスター解除したとき, the Web UI shall 当該記事を含む「お気に入り」一覧のキャッシュを無効化し、次回再取得時に当該記事が一覧から除外されている状態にする。
3. When 「お気に入り」一覧から記事をスター解除した直後にユーザーが他フィードへ切り替えてから再度「お気に入り」を選択したとき, the Web UI shall 当該記事を表示しない最新状態の一覧を再取得する。
4. When ユーザーが個別フィード画面でスター解除を行ったあとに「お気に入り」を選択したとき, the Web UI shall 当該記事を含まない最新の一覧を表示する。

### Requirement 4: 全フィード横断スター記事一覧 API の挙動

**Objective:** As フロントエンド, I want 認証済みユーザーのスター付き記事を横断取得する API があること, so that 個別フィード API を複数回呼ぶことなく一覧を構築できる

#### Acceptance Criteria

1. When 認証済みユーザーが全フィード横断スター記事一覧エンドポイント `GET /api/feeds/starred/items` を呼び出したとき, the Item API shall リクエストユーザー自身がスター付与した記事のみを応答に含める。
2. The Item API shall 当該エンドポイントの応答記事を公開日時の降順で並べる。
3. The Item API shall 当該エンドポイントの応答スキーマを、既存の単一フィード記事一覧 `GET /api/feeds/{feedID}/items` と同一形（`items` 配列・`next_cursor`・`has_more` を含む）で返す。
4. When クライアントが当該エンドポイントを `cursor` クエリパラメータなしで呼び出したとき, the Item API shall 公開日時降順の先頭ページを返却する。
5. When クライアントが当該エンドポイントに前回応答の `next_cursor` を `cursor` クエリパラメータとして指定したとき, the Item API shall 当該カーソル時点より後続のページを返却し、既存の単一フィード API と同じカーソル送り規約を維持する。
6. If 未認証のリクエストが当該エンドポイントを呼び出したとき, the Item API shall HTTP 401 Unauthorized を返却し、応答ボディに記事データを含めない。
7. If 当該エンドポイントを呼び出したユーザーがスター付与した記事を 1 件も持たないとき, the Item API shall HTTP 200 を返却し、`items` が空配列で `has_more` が `false` の応答とする。
8. If クライアントが当該エンドポイントに不正な `cursor` 値（既存の単一フィード API と同じ判定基準で解釈不能な文字列）を指定したとき, the Item API shall 既存の単一フィード API と同等のクライアントエラー応答を返却する。
9. The Item API shall 当該エンドポイントの応答に、別ユーザーが付与したスター記事を一切含めない（ユーザー境界の漏洩を防ぐ）。
10. The Item API shall 当該エンドポイントが返す記事行に、当該記事の所属フィード識別情報（フィード ID）を含めることで、フロントエンドが Requirement 2.4 のフィード識別表示を再構成できるようにする。

### Requirement 5: 既存挙動への非干渉

**Objective:** As 既存ユーザー, I want 既存のフィード別記事一覧やスター操作が本機能追加で変化しないこと, so that 本機能を使わない動線でも従来通りの体験が維持される

#### Acceptance Criteria

1. The 既存単一フィード記事一覧 API `GET /api/feeds/{feedID}/items` shall 本機能追加の前後で、リクエスト・レスポンス・並び順・カーソル仕様・フィルタ挙動を変更しない。
2. The 既存スター更新エンドポイント shall 本機能追加の前後で、リクエスト・レスポンス・状態遷移挙動を変更しない。
3. The Web UI shall 既存の個別フィード行のクリック動線、フィルタタブ（全て/未読/スター）、記事詳細展開挙動を本機能追加の前後で変更しない。

## Non-Functional Requirements

### NFR 1: 性能（応答時間の同等性）

1. While 同一ユーザーが「お気に入り」一覧の初回ページを取得している間, the Item API shall 当該リクエストの応答時間を、同一ユーザーが既存の単一フィード記事一覧（`GET /api/feeds/{feedID}/items`）に対して同件数の応答を返すときと同等以内（観測されるレスポンスタイムの差が単一フィード基準に対し有意に悪化しない範囲）に維持する。
2. The Item API shall 「お気に入り」一覧のクエリ実行において、スター付き記事を効率的に絞り込むためのインデックス（`idx_item_states_user_starred` 相当の既存部分インデックス）の前提を利用可能な状態に保つ（部分インデックスの存在を破壊しないことを含む）。

### NFR 2: 認可境界（ユーザー分離）

1. The Item API shall 「お気に入り」一覧の応答において、リクエストユーザーと異なるユーザーがスター付与した記事を、いかなる入力組合せ（`cursor` 操作・他ユーザーのスター付き状態など）でも応答に含めない。

### NFR 3: 既存応答スキーマとの整合（フロントエンド互換）

1. The Item API shall 「お気に入り」一覧の応答 JSON のフィールド名・型・並び順・空配列時の表現を、既存の単一フィード記事一覧 API の応答と区別不能（フィールド集合とセマンティクスが同一）にする。

## Out of Scope

- お気に入り記事のフォルダ分け・タグ付け・並べ替え（Issue 本文で明示的にスコープ外）。
- 既存スター更新エンドポイントの新規追加・廃止・パラメータ変更。
- スター以外の「保存」概念（後で読む等）の導入。
- 「お気に入り」一覧上での未読/全件サブフィルタ UI の追加。
- 既存の単一フィード `GET /api/feeds/{feedID}/items?filter=starred` 経路の廃止（横断 API 追加後も既存挙動を残す前提とする。経路統廃合の判断は本要件のスコープ外）。
- インデックス設計や SQL チューニングの実装手段（NFR 1 の応答時間目標を満たす方法は `design.md` の領分）。

## Open Questions

- 「お気に入り」一覧の記事行で併記するフィード識別情報の具体的表現（フィードタイトルのみ表示するか、favicon を併記するか、表示位置をどこにするか）は Requirement 2.4 で「フィード識別情報を併記する」までを確定し、最終的なビジュアル仕様は design 段階で確定する余地を残す。
- Requirement 3 のスター解除反映を「楽観的 UI 更新で即時にお気に入り一覧から消す」とするか、「キャッシュ無効化＋再取得タイミングで消す」とするかは、既存のフィード別スター切替（楽観的更新あり）との UX 整合性の観点から、最低保証（再取得時に消える）を本要件で確定し、楽観的更新を採用するか否かは design / 実装段階で確定する。
- Requirement 4.8 の「不正カーソル時のクライアントエラー応答」は、既存の単一フィード API が `cursor` を `time.Time` RFC3339(Nano) 文字列としてパースし失敗時に検証エラーを返す挙動と同一基準を採用する想定だが、横断 API のカーソル表現を将来的に変更する余地があるなら design 段階で再確認する（本要件では「既存と同等の判定基準」までを確定）。
